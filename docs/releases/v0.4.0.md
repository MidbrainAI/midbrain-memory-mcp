# midbrain-memory-mcp v0.4.0

## TLDR

v0.4.0 turns MidBrain Memory into an experience-first layer for long-running
coding agents across OpenCode, Claude Code, Codex, and NanoClaw.

- Adds Codex support.
- Adds NanoClaw support through `/add-midbrain`.
- Adds `check_session_status` so agents can detect recent work from other
  sessions before planning from stale context.
- Adds `memory_type` filtering to `memory_search`.
- Adds memory-first project rules for `AGENTS.md` and `CLAUDE.md`.
- Adds automatic procedural knowledge injection before supported user turns.
- Centralizes client adapters, key resolution, API access, and capture helpers.
- Keeps project keys safer by preserving existing keys, using `chmod 600`, and
  avoiding full key output.

Upgrade or install:

```bash
npx -y midbrain-memory-mcp@latest install
```

Then restart OpenCode, Claude Code, Codex, or the relevant NanoClaw group.

## What Changed

- Added supported setup paths for Codex and NanoClaw.
- Added memory-first project rules so agents start with project context instead
  of waiting for users to repeat it.
- Added cross-session continuity through the new `check_session_status` MCP
  tool and recency hints on memory search flows.
- Added selective procedural knowledge injection on supported user-turn capture
  paths, so learned workflows can guide the model when they are relevant.
- Reworked the package around shared client adapters, centralized key
  resolution, and shared API/capture helpers.
- Hardened installer behavior, stale hook/plugin repair, project setup, and
  package contents.

## Privacy And Automatic Capture

Installing a supported client wires local hooks or plugins that contact
`https://memory.midbrain.ai` with the configured API key. By default, those
hooks capture user prompts and assistant messages as episodic memories. Codex
also buffers tool events locally and stores one bounded tool-activity summary
per turn.

Procedural-knowledge injection searches MidBrain before supported user turns
and may prepend returned context before the model runs. User prompts and
assistant answers are stored as provided, except MidBrain-injected procedural
context is scrubbed from assistant captures. Codex tool summaries redact common
secret keys, bearer tokens, and secret-like fields, but this is not a general
redaction pass over all captured conversation text.

Capture and procedural injection are best-effort and fail open. Key-resolution,
network, capture, and procedural-search failures do not block the client turn.
The tradeoff is that a failed capture may leave no episodic memory for that
turn.

## Codex Support

MidBrain now detects and configures Codex.

The installer writes Codex MCP configuration, a Codex-specific key file, and
global capture hooks for user prompts, tool activity, and assistant output.
Project setup writes `.codex/config.toml` with `MIDBRAIN_PROJECT_DIR` so memory
search and capture can use the project-specific MidBrain agent after restart.

Codex project setup does not write project-local hooks. Capture remains global
to avoid duplicate memories when both global and project hook layers could
match the same turn. After install, restart Codex and review hook trust prompts
with `/hooks` if Codex asks.

## NanoClaw Support

MidBrain now includes a NanoClaw adapter and an installable `/add-midbrain`
skill.

Running the installer copies the skill into the NanoClaw skills directory. From
the NanoClaw directory, open Claude Code and run:

```text
/add-midbrain
```

The skill walks through selecting the target group, wiring the MidBrain MCP
server, installing `midbrain-memory-mcp@latest` in the group image, merging
Claude capture hooks into the group settings, preserving existing settings,
redacting inline keys in output, and restarting only after approval.

Manual NanoClaw MCP setup can provide search tools, but automatic episodic
capture requires the skill's direct Claude settings merge.

## Memory-First Project Setup

Project setup can now write a bounded MidBrain rules block into `AGENTS.md` and
`CLAUDE.md`.

Those rules tell agents to check session status, search memory before work that
depends on prior context, use exact-match memory tools for precise anchors,
retrieve recent episodic context when continuing work, avoid manually creating
memories, and use `memory_setup_project` instead of hand-editing key or MCP
config files.

Why this matters: memory only helps if agents remember to use it before they
act. Putting the rules in project instruction files changes memory from
something the user has to request into part of the agent's startup behavior.

Two setup flows can write the rules:

- `npx -y midbrain-memory-mcp@latest install --project <absolute-path>` writes
  them by default.
- The normal installer writes them when run from a detected project directory:
  interactive runs prompt first; non-interactive runs write by default.

Existing project instructions are preserved. MidBrain only creates or replaces
the block between:

```md
<!-- midbrain-memory-rules:start -->
<!-- midbrain-memory-rules:end -->
```

Use `--no-rules` to opt out of instruction-file writes only. It does not
disable MCP configuration, capture hooks, or procedural-knowledge injection.

## Continuity And MCP Tools

`check_session_status` lets an agent quickly detect recent activity from
another session or client, then use `get_episodic_memories_by_date` when it
needs the full context.

`memory_search` supports `memory_type` filtering: `all`, `semantic`, or
`episodic`. `memory_search`, `grep`, `list_files`, and `read_file` can append
recency hints when unseen episodic activity exists.

For clarity, the full current MCP tool surface in `0.4.0` is:

- `memory_search`
- `grep`
- `get_episodic_memories_by_date`
- `list_files`
- `read_file`
- `check_session_status`
- `memory_setup_project`

`memory_setup_project` remains the MCP path for project memory configuration.
It creates or preserves the project key, writes project-level client configs,
and tells the agent to restart the client so the new project key takes effect.
It does not write `AGENTS.md` or `CLAUDE.md`; instruction-file writes stay in
the installer flows for now.

## Selective Procedural Knowledge Injection

Supported capture paths can inject relevant procedural knowledge automatically
before a user turn.

The hook searches for matching procedures with a short timeout and prepends a
bounded context block only when useful. There is no public procedural-knowledge
MCP tool for agents to call manually, and users do not have to paste the same
workflow rules into every session.

The injection path is designed to be quiet and bounded:

- If procedural search fails or times out, the user turn continues unchanged.
- Injected context is capped at 160 characters per title, 2,000 characters per
  entry body, and 6,000 characters total.
- Marker-like text is escaped.
- Trusted injected blocks include signed `ctx-meta nonce` metadata so
  user-authored marker examples cannot spoof deduplication.
- Assistant capture strips injected PK blocks before storing assistant text
  back into episodic memory.

## Capture And Architecture

v0.4.0 reorganizes the package around shared client adapters and shared runtime
helpers.

The practical effect is simpler client support and more consistent behavior
across OpenCode, Claude Code, Codex, and NanoClaw. MCP tools, hooks, and
plugins now resolve keys through the same client layer, use the same API
helper, and share procedural-knowledge formatting and scrubbing logic.

Capture behavior is also more complete:

- OpenCode captures user messages and completed assistant messages.
- Claude Code captures user prompts and assistant stops.
- Codex captures user prompts, assistant output, and bounded tool activity
  summaries.
- Captured memories include client metadata, which makes cross-client
  continuity easier to reason about.

On MCP server startup, MidBrain checks installed hook/plugin paths and may
automatically update stale paths in `~/.claude/settings.json`,
`~/.codex/hooks.json`, or `~/.config/opencode/plugins/` to point at the current
package location. This repair is fire-and-forget and not interactive; failures
are logged and do not stop MCP startup.

## Upgrade Notes

Prerequisites are unchanged: Node 20+, a MidBrain API key, and at least one
supported client installed.

Upgrade or install:

```bash
npx -y midbrain-memory-mcp@latest install
npx -y midbrain-memory-mcp@latest --version
```

The version check should print `0.4.0`.

If an MCP config is pinned to `midbrain-memory-mcp@0.3.2`, update it to
`midbrain-memory-mcp@0.4.0` or, preferably, `midbrain-memory-mcp@latest`.
Check global and project-level configs for OpenCode, Claude Code, and Codex.

Project setup:

```bash
npx -y midbrain-memory-mcp@latest install --project /absolute/path/to/project
```

Restart after setup:

- OpenCode and Claude Code: restart the client.
- Codex: restart Codex and review hook trust prompts with `/hooks` if prompted.
- NanoClaw: run the installer, open Claude Code from the NanoClaw directory,
  run `/add-midbrain`, choose the group, and approve config/restart prompts.

## Compatibility And Trust Notes

No intentional breaking change is expected for existing OpenCode or Claude Code
users.

Important behavior changes:

- Project CLI setup now writes `AGENTS.md` and `CLAUDE.md` by default. Use
  `--no-rules` to opt out.
- `--no-rules` is an instruction-file opt-out only. To run search-only, do not
  install capture hooks/plugins for that client, or remove/disable the MidBrain
  hook/plugin entries after installation.
- The `memory_setup_project` MCP tool remains config/key-only and does not
  write instruction files.
- Project memory changes require a client restart because keys are resolved at
  startup.
- Existing project keys are preserved; setup does not overwrite credentials.
- Key files written by the installer use `chmod 600`, and installer output
  reports key locations/status without printing full API keys.
- Empty key files and unreadable key files are treated as hard errors instead
  of silently falling through.
- Inline NanoClaw hook keys are redacted from output by the skill instructions.

The published package includes the runtime server, installer, shared client
adapters, hooks, NanoClaw skill, built OpenCode plugin bundle, and README.
Tests, task files, local configs, and key files are excluded from the package.
