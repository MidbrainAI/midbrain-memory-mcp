# midbrain-memory-mcp v0.4.2

## TLDR

v0.4.2 is a focused Codex reliability release.

- Codex trusts hooks by command identity.
- Old MidBrain Codex hook commands could change when npm cache paths, package
  paths, or Node/Homebrew paths changed.
- If Codex required `/hooks` re-approval and it was missed, capture could
  silently stop.
- Codex hooks now use a stable local shim:

  ```text
  ~/.midbrain/bin/codex-hook user
  ~/.midbrain/bin/codex-hook tool
  ~/.midbrain/bin/codex-hook assistant
  ```

- Codex users may need to approve MidBrain once with `/hooks` after v0.4.2.
- Normal package, npm cache, and Node updates should stop forcing repeated
  approval after that.
- Breaking changes: None.

## Overview

v0.4.2 fixes a trust stability problem introduced by the Codex capture path in
the v0.4.x line. Codex App trusts hooks by command identity. Before this
release, MidBrain's Codex hooks pointed directly at package-cache scripts and
the local Node executable. Normal environment changes, including npm cache
movement, package reinstall paths, or Homebrew Node updates, could change the
hook command that Codex had previously trusted.

When that command identity changed, Codex could require a fresh `/hooks`
approval. If the approval was missed, MidBrain memory capture could stop
silently.

v0.4.2 moves Codex capture through a stable local shim:

```text
~/.midbrain/bin/codex-hook user
~/.midbrain/bin/codex-hook tool
~/.midbrain/bin/codex-hook assistant
```

After upgrading, Codex users may need to approve the migrated MidBrain hooks
once with `/hooks`. After that one-time approval, normal MidBrain package
updates, npm cache changes, and Node path changes should no longer force
repeated hook re-approval.

## What Changed

- Added a stable Codex hook shim at `~/.midbrain/bin/codex-hook`.
- Updated installer-written Codex hooks to call the shim instead of direct
  `plugins/codex/capture-*.mjs` script paths.
- Added CLI dispatch for Codex hook entry points:
  - `npx -y midbrain-memory-mcp@latest hook codex user`
  - `npx -y midbrain-memory-mcp@latest hook codex tool`
  - `npx -y midbrain-memory-mcp@latest hook codex assistant`
- Migrated stale legacy/direct MidBrain Codex hooks to the stable shim.
- Preserved foreign, non-MidBrain hooks in `~/.codex/hooks.json`.
- Hardened startup repair so stale MidBrain Codex hooks are repaired
  conservatively without auto-installing capture hooks where none existed.
- Updated README and AGENTS documentation with the one-time `/hooks` approval
  note and the stable-shim trust model.
- Added regression coverage for stable commands, migration, startup repair,
  CLI hook dispatch, foreign hook preservation, and docs.

## Why It Matters

The v0.4.0 release added Codex support, including user, tool, and assistant
capture. That made Codex part of MidBrain's cross-session memory loop, but it
also exposed a practical trust issue: command paths are often not stable across
JavaScript package updates.

Direct hook commands can include paths like npm cache directories, package
install locations, or Homebrew-managed Node binaries. Those are implementation
details, not stable trust anchors. When they change, Codex is right to ask the
user to review the hook again, but the operational result is fragile: memory can
appear installed while capture has stopped waiting for approval.

The shim gives Codex one stable local command to trust. The shim then resolves
the latest MidBrain package internally with
`npx -y midbrain-memory-mcp@latest hook codex ...`.

The result is less silent capture drift after routine updates.

## Codex Upgrade Note

After installing v0.4.2, Codex users should restart Codex and run:

```text
/hooks
```

Approve the MidBrain hook commands if prompted. The commands should point at:

```text
~/.midbrain/bin/codex-hook
```

This approval may be required once because the trusted command is intentionally
changing from direct package-cache paths to the stable shim.

## Trust Tradeoff

The trust model changes deliberately.

Before v0.4.2, approving a MidBrain Codex hook often meant approving a specific
direct command path. That path could be tied to a particular npm cache location,
package install path, or Node binary path. It was precise, but too brittle for a
package that is normally installed and upgraded through `npx` and `@latest`.

In v0.4.2, approving the MidBrain Codex hook means trusting the stable local
shim:

```text
~/.midbrain/bin/codex-hook
```

That shim calls `midbrain-memory-mcp@latest` for the Codex hook dispatcher. This
is more stable operationally, but it means the approval is for MidBrain's
auto-updating package command rather than for one immutable npm cache file.

This is the intended tradeoff: fewer repeated approvals and fewer silent capture
outages, with a clearer trust boundary around the MidBrain package command.

## Compatibility

No intentional breaking change is expected for OpenCode, Claude Code, NanoClaw,
or MCP tool users.

Codex behavior changes are limited to MidBrain hook installation, migration, and
repair:

- Existing foreign hooks are preserved.
- Existing legacy MidBrain Codex hooks are replaced with shim commands.
- If no MidBrain Codex hooks are installed, startup repair does not install new
  capture hooks automatically.
- `Stop` and `PostToolUse` hook paths continue to return `{}` on failure-safe
  paths as required by Codex hook behavior.
