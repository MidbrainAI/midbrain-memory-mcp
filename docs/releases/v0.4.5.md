# v0.4.5

v0.4.5 makes episodic memory capture more resilient when the MidBrain API or
network is temporarily unavailable.

## TLDR

- Failed episodic writes are cached locally and retried after a later
  successful write.
- Cache files are scoped by API key and project so entries do not flush across
  accounts or workspaces.
- Concurrent cache flushes now use explicit ownership so one process cannot
  delete another process's in-flight batch.
- Interrupted `.processing` batches can be recovered on a later flush.
- Cache tests cover outage simulation, scoped isolation, permissions, survivor
  preservation, and concurrent flush ownership.

## Why

Episodic capture is only useful if short outages do not create quiet memory
gaps. Before this release, a failed `storeEpisodic` call could be lost when the
API was down, a user was offline, or a capture hook exited before the request
could succeed.

v0.4.5 adds a small file-backed retry path for failed episodic writes. The
implementation is intentionally local and dependency-free, and it keeps
credential boundaries intact by scoping cache files without storing raw API
keys.

## Changes

- Added a file-backed NDJSON cache for failed episodic memory writes.
- Added `MIDBRAIN_SIMULATE_OFFLINE=1` as a test/debug path that exercises the
  cache-on-failure flow without making a network request.
- Scoped cache paths by project and API-key fingerprint so entries from one
  key or workspace cannot be flushed by another.
- Stored cache files under a private cache directory with restrictive
  permissions where the platform supports them.
- Added a live-to-processing flush handoff so interrupted flush batches can be
  retried later.
- Preserved survivor entries and concurrent live appends when only part of a
  cached batch flushes successfully.
- Added explicit processing-batch ownership so losing concurrent flushers do
  not delete or merge another process's claimed batch.

## Compatibility

The MCP tool surface and install flow are unchanged. Successful episodic writes
still post directly to the MidBrain API.

The new cache is best-effort durability for failed episodic writes. It does not
change API key resolution, generated client configs, or procedural-knowledge
behavior.

## Testing

- Episodic cache unit tests for append, flush, recovery, survivors, scoped
  isolation, and permissions.
- `MidbrainApi.storeEpisodic` tests for failed-write caching, retry behavior,
  and `MIDBRAIN_SIMULATE_OFFLINE=1`.
- Codex hook tests for offline capture, restart recovery, and scoped cache
  flushing.
- Full `npm run check` and package dry-run before publish.

## Breaking Changes

None.
