# v0.4.4

v0.4.4 ships the browser/device-code installer auth flow from PR #25, so a
first-time user can sign in at memory.midbrain.ai, run the installer, and get a
working memory agent without manually copying an API key.

## TLDR

- First-run global installs can use browser-based auth through
  memory.midbrain.ai.
- The installer can create or select an agent, create an API key, write key
  files, and patch supported client configs from one command.
- `--no-login` now correctly skips browser/device auth and stays on the manual
  key fallback path.
- The README Quick Start now reflects the sign-in, install, restart flow.
- Device-auth and installer-boundary tests cover the new FTUX path.

## Why

The old first-time setup path asked users to understand agents, API keys, key
files, config files, and client wiring before they could see memory working.
That was too much ceremony for closed-alpha onboarding and low-friction design
partner setup.

v0.4.4 narrows that path. The intended default is now: sign in, run
`npx midbrain-memory-mcp install`, restart the client, and verify the memory
tools are available.

## Changes

- Added a device-code authorization flow for the CLI installer.
- Interactive global installs with no existing key now offer browser-based
  authentication by default.
- The installer opens memory.midbrain.ai, waits for authorization, lets the user
  create or select an agent, finalizes an agent API key, and writes the returned
  key through the existing secure key paths.
- Existing key files still skip login unless `--login` is used explicitly.
- `--no-login` now suppresses browser/device auth and uses the existing manual
  key fallback. If both `--login` and `--no-login` are present, `--no-login`
  wins.
- The README Quick Start now starts with signing in at memory.midbrain.ai and
  documents manual key paste only as the `--no-login` fallback.
- Added tests for device auth polling/finalization, existing-agent selection,
  no-agent creation, expired/failed auth responses, default installer login,
  existing-key skips, forced login, and `--no-login` precedence.

## Compatibility

Existing users with key files keep the same behavior by default. Non-interactive
installs still require an existing key file or `MIDBRAIN_API_KEY`.

Manual setup remains available:

```sh
npx midbrain-memory-mcp install --no-login
```

Project setup behavior is unchanged.

## Testing

- Device-code authorization flow unit tests.
- Installer global auth and manual fallback tests.
- Existing-key, forced-login, and flag-precedence tests.
- README onboarding update.
- Full `npm run check` and package dry-run before publish.

## Breaking Changes

None. The new browser auth flow is the default first-run path for interactive
global installs without an existing key, and manual key setup remains available.
