# midbrain-memory-mcp v0.4.2

v0.4.2 is a focused Codex reliability release.

Codex App trusts hook commands by command identity. In earlier v0.4.x builds,
MidBrain's Codex hooks could point at direct package-cache scripts or local Node
paths. Routine changes such as npm cache movement, package updates, or Homebrew
Node updates could change those commands and require a fresh Codex `/hooks`
approval. If that approval was missed, memory capture could silently stop.

This release moves Codex capture to a stable local shim:

```text
~/.midbrain/bin/codex-hook user
~/.midbrain/bin/codex-hook tool
~/.midbrain/bin/codex-hook assistant
```

After upgrading, Codex users may need to approve MidBrain once with `/hooks`.
After that, normal MidBrain package updates, npm cache changes, and Node path
changes should stop forcing repeated re-approval.

## Highlights

- Stable Codex hook shim at `~/.midbrain/bin/codex-hook`.
- Codex hook CLI dispatch for `hook codex user`, `hook codex tool`, and
  `hook codex assistant`.
- Migration of stale legacy/direct MidBrain Codex hooks.
- Preservation of foreign hooks in `~/.codex/hooks.json`.
- Conservative startup repair for stale MidBrain Codex hooks.
- README and AGENTS documentation for the one-time `/hooks` approval and shim
  trust model.
- Regression tests for migration, repair, stable commands, CLI dispatch, docs,
  and hook preservation.

## Trust Note

The shim changes the trust boundary. Approving the new Codex hook means trusting
MidBrain's stable local shim, which resolves `midbrain-memory-mcp@latest`
internally, rather than trusting one direct npm cache file or one Node binary
path.

That tradeoff is intentional: one clear MidBrain command to approve, fewer
repeated approvals after routine updates, and less risk that capture silently
stops because the underlying package path changed.

Breaking changes: None.
